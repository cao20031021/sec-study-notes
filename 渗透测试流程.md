## 前言
渗透测试是网络安全运维、安全评估的核心技术手段，是在合法授权前提下，模拟黑客攻击思路，主动发现网络、服务器、Web 应用、设备存在的安全漏洞，从而修复风险、提升资产安全防护能力的合规测试行为。
标准完整的渗透测试分为五大核心阶段：
获得授权 → 信息收集 → 漏洞利用 → 清除痕迹 → 编写测试报告
本文将逐阶段拆解原理、操作流程、核心要点、合规规范，适配企业渗透测试、护网演练、安全实训场景，可直接用于 GitHub 开源学习。
## 一、获得授权（渗透测试前置核心阶段）
### 1.1 阶段核心意义
授权是渗透测试的唯一合法前提，无授权的一切测试行为均为非法入侵。本阶段是区分「合规安全测试」和「网络攻击」的核心边界，也是 GitHub 开源项目必须标注的第一合规要素。
### 1.2 授权获取规范流程
#### 1.2.1 书面授权确认
必须获取资产归属方出具的正式渗透测试授权书，授权内容必须包含：
被测资产完整范围：IP 地址、域名、服务器端口、Web 系统、内网设备清单
授权测试时间：明确起止日期，禁止超期测试
授权测试权限：是否允许端口扫描、漏洞利用、权限提升、数据读取测试
免责声明：资产方自愿接受安全测试，承担测试合规授权责任
双方签字 / 盖章、有效日期
#### 1.2.2 测试范围界定
严格遵循最小授权原则
禁止测试授权范围外的 IP、域名、子系统、内网设备
禁止对第三方关联资产、公共网络设备进行扫描测试
禁止在测试中干扰业务正常运行、删除业务数据、篡改核心配置
#### 1.2.3 应急沟通机制建立
测试前与资产方安全负责人建立实时沟通渠道，针对：
测试导致业务卡顿、宕机的应急终止
高危漏洞实时上报
突发风险紧急处理
### 1.3 本阶段交付物
《渗透测试正式授权书》《测试资产范围清单》
## 二、信息收集（渗透测试基础阶段，决定测试成功率）
信息收集是渗透测试最耗时、最关键的基础环节，收集信息越全面，后续漏洞挖掘成功率越高。本阶段全程为无危害探测，仅做信息枚举、资产梳理，不执行任何攻击、写入、篡改操作。
### 2.1 信息收集分类
分为：公开信息收集（外网被动收集）、主动资产探测（外网主动扫描）、内网信息收集（授权内网环境）
### 2.2 被动信息收集（不触碰目标资产，无日志留存）
无需发送数据包至目标服务器，通过公开互联网资源获取资产信息，安全无痕迹。
#### 2.2.1 域名与子域名收集
收集目标主域名、全部子域名、备案信息、域名注册人、注册时间
工具 / 平台：Whois 查询、爱站、站长工具、Subdomain、OneForAll
核心目的：扩大测试资产范围，发现边缘薄弱子系统（后台、测试站、备用系统）
#### 2.2.2 企业公开信息收集
企业官网、招聘信息、运维邮箱、员工信息、办公地址、业务架构
核心目的：用于社工信息整理、弱口令爆破字典构建
#### 2.2.3 历史漏洞与源码信息收集
Github、Gitee 公开泄露源码、配置文件、账号密码
漏洞平台历史漏洞记录、公开 CVE 适配情况
核心目的：发现历史遗留未修复漏洞、源码泄露、密钥泄露
### 2.3 主动信息收集（合规扫描，留存合法测试日志）
在授权范围内，主动向目标资产发送数据包，探测资产状态。
#### 2.3.1 存活主机探测
探测目标网段内存活的服务器、主机、网络设备
工具：ping、arp-scan、Nmap 存活扫描
#### 2.3.2 端口与服务探测
扫描目标开放端口、对应服务版本、运行程序、操作系统版本
常见重点端口：80/443（Web 服务）、22（SSH）、3306（MySQL）、3389（远程桌面）
工具：Nmap、Masscan
核心目的：根据服务版本匹配对应已知漏洞
#### 2.3.3 Web 架构信息收集
探测 Web 中间件：Apache、Nginx、Tomcat、IIS
探测编程语言：PHP、Java、Python、ASP
探测 CMS 系统：织梦、帝国、ThinkPHP、SpringBoot
探测 WAF 防护设备及防护策略
#### 2.3.4 目录与文件扫描
扫描网站后台目录、备份文件、配置文件、日志文件、测试文件
工具：Dirsearch、Dirmap、御剑
### 2.4 本阶段输出结果
《目标资产信息梳理表》，包含：存活 IP、开放端口、服务版本、域名清单、Web 架构、敏感目录、公开泄露信息汇总
## 三、漏洞利用（渗透测试核心攻击阶段）
在合法授权、不破坏业务的前提下，针对信息收集阶段发现的风险点，进行漏洞验证与利用，获取系统权限、验证风险危害等级。
### 3.1 漏洞筛选原则
优先验证高危可利用漏洞：远程代码执行、SQL 注入、文件上传漏洞、权限绕过、未授权访问
规避破坏性漏洞：格式化磁盘、批量删库、系统崩溃类漏洞，仅做原理验证
严格禁止：勒索加密、数据窃取、批量遍历用户隐私数据
### 3.2 标准漏洞利用流程
#### 3.2.1 漏洞验证
针对扫描出的漏洞 POC，进行合法性验证，确认漏洞真实存在、可复现，排除误报。
步骤 1：匹配服务版本对应 CVE 漏洞编号
步骤 2：使用安全 POC 进行漏洞检测
步骤 3：记录漏洞触发路径、影响范围、危害程度
#### 3.2.2 漏洞利用提权
漏洞验证成功后，根据测试授权进行可控利用：
Web 漏洞：获取网站后台权限、webshell 权限、读取配置文件
服务器漏洞：获取普通用户权限，尝试本地提权至系统权限
数据库漏洞：验证数据库弱口令、是否可外连、数据泄露风险
#### 3.2.3 内网横向移动（授权允许前提下）
获取单点权限后，探测内网网段、内网主机，发现内网横向漏洞，评估整体内网安全架构风险。
### 3.3 漏洞利用红线规范
禁止篡改、删除、新增业务核心数据
禁止搭建持久化后门、隐蔽后门用于长期控制
禁止对外传播测试过程中获取的任何企业数据
业务高峰期禁止执行高负载漏洞利用操作，避免业务瘫痪
### 3.4 本阶段输出结果
《漏洞复现记录表》，包含：漏洞名称、CVE 编号、影响资产、复现步骤、漏洞危害、权限获取结果
## 四、清除痕迹（合规测试收尾关键阶段）
渗透测试完成、漏洞验证结束后，必须彻底清除所有测试痕迹，保证目标资产运维环境干净、无遗留后门、无恶意日志、无残留文件，避免给客户带来二次安全风险。
本阶段是专业渗透测试与恶意入侵的重要区分点。
### 4.1 日志痕迹清除
#### 4.1.1 系统日志清除
Windows：安全日志、系统日志、应用程序日志、远程登录日志
Linux：/var/log/ 系列登录日志、操作日志、bash 历史命令
#### 4.1.2 Web 日志清除
清除 Nginx/Apache/Tomcat 访问日志、错误日志中所有测试扫描、漏洞利用、访问记录。
### 4.2 测试文件与后门清除
删除测试过程上传的所有木马、webshell、测试脚本、POC 脚本
删除临时创建的用户、权限、计划任务、持久化服务
删除本地缓存的密码文件、配置文件、导出的日志文件
### 4.3 环境复原操作
恢复测试过程修改的配置文件、权限配置、端口配置
关闭测试期间开启的临时端口、共享权限
撤销所有临时授权、临时权限提升操作
### 4.4 痕迹检查核验
清除完成后，二次核验：无遗留文件、无遗留账号、无遗留日志、无配置篡改，确保资产恢复测试前原始状态。
## 五、编写渗透测试报告（最终交付阶段）
渗透测试报告是整个测试项目的最终交付成果，也是 GitHub 开源展示的核心文档。报告要求客观、真实、完整、可落地，包含风险评估、修复方案、整改建议。
### 5.1 报告标准结构（通用企业版）
#### 5.1.1 测试概述
测试背景、测试目的、测试依据（网络安全法、等保 2.0）
测试时间、测试人员、授权范围、测试环境
#### 5.1.2 测试流程说明
简述「授权确认→信息收集→漏洞验证→痕迹清除」完整测试过程
#### 5.1.3 资产梳理清单
汇总所有被测资产、IP、域名、服务、系统信息
#### 5.1.4 漏洞详情汇总
按风险等级分级：高危漏洞、中危漏洞、低危漏洞、风险提示
每条漏洞包含：
漏洞名称、漏洞编号、影响资产
漏洞原理、复现步骤、漏洞截图
漏洞危害分析
#### 5.1.5 漏洞修复方案
针对每一个漏洞提供可直接落地的修复步骤、临时应急方案、长期防护方案
#### 5.1.6 整体安全风险评估
分析目标资产整体安全现状、架构短板、防护缺失点
#### 5.1.7 安全加固建议
服务器、Web 应用、内网架构、账号口令、WAF 防护、运维规范全方位加固建议
### 5.2 报告编写规范
语言正式、客观，不夸大、不缩小漏洞风险
所有漏洞均附带复现截图、验证证据
修复方案通俗易懂，适配运维人员直接操作
禁止在报告中留存可直接利用的恶意后门、完整攻击 Payload
### 5.3 最终交付物
《XX 资产渗透测试安全评估报告》

## 七、总结
完整的合规渗透测试，是一套合法、规范、可控、可溯源、可收尾的标准化安全评估流程。
核心逻辑闭环：
合法授权（定边界）→全面收集（找风险）→可控利用（验危害）→彻底清痕（保安全）→书面报告（落整改）
所有网络安全测试从业者，必须坚守「未授权不测试、授权内操作、测试后清零」的职业底线，遵守国家网络安全法律法规，杜绝一切黑灰产行为。
